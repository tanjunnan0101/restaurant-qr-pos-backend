1. Executive Summary
This document defines the first complete development specification for a deployable restaurant QR ordering POS platform. The 
system is designed to work on tablets, phones, laptops, and browser-based POS terminals without requiring dedicated proprietary 
hardware.
Core operating flow
Customer scans table QR -> browses menu -> selects modifiers/add-ons -> pays in browser by Stripe card or PayNow -> webhook 
confirms payment -> order appears in POS and KDS -> receipt/kitchen ticket prints -> inventory deducts -> owner sees sales, 
payment, and inventory data in the backend dashboard.
1.1 Product Positioning
 Restaurant-first POS, not generic retail POS.
 Mobile-first QR ordering for dine-in, takeaway, pickup, events, and food court style use cases.
 Designed to reduce cashier workload by allowing customers to order and pay from their own phones.
 Web-first deployment so the system can run on iPad, Android tablet, phone, desktop browser, touchscreen POS, and kitchen 
tablet.
 Singapore-ready payment support including Stripe card, Stripe PayNow, manual PayNow, SGQR/manual settlement workflows, 
GST, service charge, and PayNow reconciliation.
2. Goals and Non-Goals
2.1 Main Goals
1. Reduce cashier manpower by enabling self-ordering and self-payment through table QR codes.
2. Allow restaurants to deploy the POS on common tablets and handphones instead of expensive fixed terminals.
3. Support restaurant-grade order management, kitchen routing, receipt printing, payment tracking, and sales reporting.
4. Provide a backend owner dashboard for sales, inventory, payments, refunds, PayNow status, GST, service charge, and outlet 
performance.
5. Support multi-outlet rollouts where HQ can manage menus, prices, reports, users, and outlet permissions.
6. Create a modular architecture that can later support loyalty, delivery, accounting, AI reporting, scheduling, and inventory 
automation.
2.2 Non-Goals for V1
 Do not prioritise retail barcode POS over restaurant QR ordering.
 Do not build full native iOS/Android apps before the web/PWA product is stable.
 Do not build deep enterprise accounting in V1; provide exports and integration-ready data instead.
 Do not require customers to create accounts before ordering.
 Do not depend on one proprietary printer method; build printer abstraction from the start.

30.2 Database Design Principles
 Use UUID primary keys for all public-facing entities.
 Every tenant-owned table must include company_id and usually outlet_id for tenant isolation and filtering.
 Store all money values in integer cents, not floating point decimals. Example: S$12.90 = 1290.
 Use soft deletes with deleted_at for business records such as menu items, users, printers, and inventory items.
 Use immutable historical snapshots for paid orders, prices, GST, service charge, and modifiers. A paid receipt must not change 
when an admin edits a menu later.
 Use ledger tables for stock and payments. Do not rely only on current_balance fields.
 Use database constraints for uniqueness, foreign keys, and allowed status values where possible.
 Use idempotency keys for payment creation, order submission, webhook processing, and print job creation.
 Every high-risk action must write to audit_logs inside the same transaction as the action.
 Indexes must support live POS/KDS screens, reporting filters, and webhook lookup performance.

30.8 Status State Machines
Important implementation rule
State changes must happen through service methods, not direct controller/database updates. Every transition should validate 
permissions, current status, allowed next status, payment status, print/KDS side effects, and audit requirements.
State machine Allowed transition Side effect
Order draft -> pending_payment Order created, price snapshot locked for 
payment attempt.
Order pending_payment -> payment_processing Stripe Checkout/PaymentIntent created.
Order payment_processing -> paid Webhook confirmed. Create KDS ticket and 
print jobs.
Order paid -> sent_to_kitchen Kitchen ticket dispatched.
Order sent_to_kitchen -> preparing KDS station starts work.
Order preparing -> ready KDS item/order bumped. Notify POS.
Order ready -> served Waiter marks served.
Order served -> completed Table session can close if fully paid.
Exception pending_payment -> cancelled No payment captured; release table/cart.
Exception paid -> partially_refunded/refunded Create refund row, reverse stock if 
configured, update reports.
Exception sent_to_kitchen/preparing -> voided Requires manager PIN, cancellation print 
ticket, audit log.
Payment state Allowed transition Rule
created -> pending Payment attempt created before provider 
redirect.
Amount must match server-calculated total.
pending -> processing Customer is completing payment. Show payment processing UI.
processing -> succeeded Trusted webhook confirms success. Mark order paid idempotently.
processing -> failed Provider failure/timeout. Allow retry or cancel.
succeeded -> partially_refunded/refunded Refund API/manual refund completed. Requires approval and audit.
manual_verification_required -> succeeded Staff verifies manual PayNow/SGQR. Requires permission and optional manager 
PIN.
Print job state Allowed transition Rule
queued -> sending Dispatcher/agent starts job. Lock job to avoid duplicate print.
sending -> printed Printer confirms success or agent returns OK. Store printed_at.
sending -> failed Printer/agent error. Store error and schedule retry.
failed -> retrying -> sending Retry count not exceeded. Backoff schedule.
failed -> cancelled Staff cancels failed job. Audit if ticket/receipt important.
printed -> reprinted Staff reprints. Create linked reprint job and audit log.

30.12 Printer Architecture
Printer flow
Print flow:
1. Order/payment event creates print_jobs rows.
2. print_jobs stores template + payload_json + target printer_id.
3. Print dispatcher finds queued jobs.
4. If printer connection_type = epson_epos, send through ePOS path.
5. If connection_type = escpos_bridge, local printer agent polls/pulls or receives job.
6. Agent/printer returns printed/failed status.
7. Backend updates print_jobs and print_job_attempts.
8. Failed jobs retry using backoff; staff dashboard shows failure.

1 5 seconds Retry same printer.

75.4 Non-Negotiable Engineering Rules
 All money values must be stored and calculated in integer cents. Do not use floating point for prices, GST, service charge or refunds.
 The frontend must never be trusted for final price, tax, discount, payment or order state. Backend recalculates and validates every 
checkout.
 Stripe webhooks are the source of truth for online payment success. Redirect/success pages are user experience only.
 Use idempotency keys for order submission, payment creation, webhook processing, refund creation and print job creation.
 Paid QR orders are immutable. Changes must be represented as add-on orders, adjustments, voids or refunds with audit trails.
 All high-risk actions require role permission and audit log: voids, refunds, manual PayNow verification, price override, cash variance, 
user/role changes.
 Every tenant-owned table must be scoped by company_id and, where relevant, outlet_id. No cross-outlet data leakage is acceptable.
 Every print job must have a persisted status and retry path. Silent print failure is a no-go for pilot launch.
 KDS and POS must show online/offline/device health states. Do not hide connectivity problems from operations staff.
 Use feature flags for V1.1/V2 features such as customer display, loyalty, advanced tabs, AI upsell and full offline mode.

76.1 Employee Clock-In / Clock-Out Module
The POS iPad must support an employee attendance flow so staff can clock in and clock out directly from the restaurant POS device. 
Each attendance action must be tagged to the employee, outlet, device, photo proof, and a trusted server-side timestamp.
Requirement Development specification
Clock-in entry point Staff POS login screen and POS side menu must include a Clock In / 
Clock Out action.
Employee identification Employee selects their profile or enters staff PIN. Optional future 
biometric/face matching must not be required for V1.7.
Photo capture iPad camera opens and captures a live photo at clock-in and clock-out. 
Upload the image to object storage and store the image URL/reference 
in attendance records.
Timestamp source Use backend server timestamp as the source of truth. Client/iPad time 
is display-only and must not determine payable hours.
Outlet/device tagging Attendance record stores company_id, outlet_id, POS device_id, 
browser/device info, IP address, and optional location metadata if 
enabled.
Automatic hours calculation When employee clocks out, system calculates total worked 
minutes/hours from clock-in to clock-out, less approved unpaid breaks if 
the break module is enabled.
Manager review Manager can review, approve, adjust, or flag attendance records with 
reason and audit log.
Payroll export Approved timesheets can be exported by date range, employee, outlet, 
and role for payroll processing.
76.2 Attendance User Flow
77. Employee taps Clock In on POS iPad.
78. Employee selects name or enters staff PIN.
79. System requests iPad camera permission if not already granted.
80. Employee takes a live photo; retake is allowed before submission.
81. Backend creates attendance_session with status clocked_in and server-side clock_in_at timestamp.
82. Employee works shift normally; POS can show Current Shift: clocked in at HH:MM.
83. Employee taps Clock Out at end of shift.
84. Employee takes clock-out photo; backend records server-side clock_out_at timestamp.
85. System calculates gross_minutes, unpaid_break_minutes, net_paid_minutes, and flags overtime/missing breaks if rules apply.
86. Manager reviews daily attendance and exports approved records for payroll.
